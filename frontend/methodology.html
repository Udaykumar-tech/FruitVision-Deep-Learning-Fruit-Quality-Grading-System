Section 01

Problem & Scope

The objective is to develop a robust, non-destructive system that classifies five common fruits (Apple, Banana, Grape, Mango, Orange) into three biological states: Fresh, Rotten, and Formalin-Mixed. Five fruits times three conditions yields 15 distinct classes.

The Formalin Challenge

The Formalin-Mixed class presents a unique challenge: chemical preservation masks the natural decay process. Fruits soaked in 10% formalin for 30 minutes and photographed 48 hours later look visually indistinguishable from genuinely fresh fruit to a standard convolutional network. The discriminative signal is in subtle skin-pore patterns, an unnatural sheen, and the absence of expected aging markers — all of which require the model to attend to fine-grained surface textures rather than holistic fruit shape or color.

Why this matters

Formalin (formaldehyde) is illegal as a food preservative in most jurisdictions. Vendors sometimes apply it to extend shelf life, exposing consumers to a known carcinogen. Detection is currently performed via chemical lab tests — slow, destructive, expensive, and unscalable. A non-destructive image-based detector running on a phone or edge device would enable point-of-purchase verification at scale.

Project goals

  • Multi-class classification — 15 classes covering 5 fruits × 3 conditions.
  • High Formalin recall — missing chemically-treated fruit is the worst error mode.
  • Background invariance — must work across studio shots, real markets, kitchens.
  • Explainability — Grad-CAM must show the model attending to fruit, not props.
  • Deployment readiness — at least one model viable for mobile/edge inference.

The formalin protocol

Formalin samples were generated by soaking fruits in 10% formalin solution for 30 minutes, then photographing them 48 hours later. This delay is critical — it gives the chemical effect time to manifest as the "unnatural perfection" or subtle sheen that becomes the primary discriminative signal. Earlier or later capture windows produce visually different effects.

Why this dataset is hard

  • Subtle inter-class differences — Fresh and Formalin-mixed share most visual properties.
  • Inter-fruit visual diversity — Grape (small clusters) and Mango (large elongated) have wildly different shape priors.
  • Photo session bias — multiple shots per fruit lead to near-duplicates.
  • Background uniformity — most photos have similar studio backgrounds, creating shortcut-learning risk.

Section 03

Data Preparation

Raw images cannot be trained on directly. We perform a multi-stage preparation pipeline: integrity audit, segmentation mask generation, and stratified splitting.

01
Dataset Audit

Indexed all 10,154 files; verified zero corruption; confirmed all images are RGB JPEG. File sizes range 20 KB – 3.7 MB with 90 KB median, totaling 0.9 GB on disk.

02
Mask Generation with MobileSAM

For each image, we run MobileSAM (a lightweight Segment Anything variant) with a center-point prompt to extract a pixel-perfect binary mask of the fruit region. A grid-prompt fallback fires when the area ratio is degenerate (<3% or >97%). This produces ~9,800 high-quality masks; the rest are filtered by area sanity checks.

03
Background Image Bank

We pull ~880 indoor scene images from the MIT Indoor 67 dataset on Hugging Face — 12 fruit-photography-relevant categories: kitchen, restaurant_kitchen, pantry, deli, bakery, grocerystore, dining_room, restaurant, buffet, mall, warehouse, closet. These are kept GPU-resident as a uint8 tensor for fast augmentation.

04
Group-Aware Train/Val/Test Split

The most important preparation step. We use perceptual hash clustering to identify near-duplicate photo sessions, then split entire clusters into either train, val, or test. This prevents the leakage problem detailed in section 04 and yields the final ratios: 70 % / 15 % / 15 %.

05
RAM Cache

All 9,727 valid images and their masks are decoded once and held as uint8 NumPy arrays in RAM. Total cache footprint: 2.55 GB at 256×256. This eliminates disk I/O during training — every batch is built directly from RAM, then transferred to GPU.

Section 04 · Critical Course Correction

The Leakage Problem We Caught Mid-Project

Our first model trained with a standard random stratified split achieved 99.73% test accuracy with 100% Formalin recall — numbers that looked too good to be true. They were.

When we computed Grad-CAM heatmaps to verify the model was looking at fruit, the focus lift was 0.73× — worse than random. The model was achieving 99% by classifying based on background lighting, shadow direction, and table textures specific to each photo session. Multiple shots from the same session had landed in both train and test sets, letting the model memorize session-level cues.

Naive Random Split
10,154 photos randomly assigned to train/val/test with class stratification. Photos from the same fruit's photo session can land in different splits.

Result: 8.4% near-duplicate leakage between train and test. Test accuracy meaningless.
Group-Aware Split
Cluster near-duplicates via 256-bit perceptual hash + Hamming distance ≤ 12. Whole clusters go to one split via sklearn.GroupShuffleSplit.

Result: zero group overlap. Test accuracy honest. Final numbers reflect actual generalization.

The perceptual hash clustering algorithm

Perceptual Hash (16-bit difference hash)

$$ H(I) = \{\, \mathbb{1}[g(x, y+1) > g(x, y)] \,\}_{x=1..16, y=1..16} $$

Where \(g(x,y)\) is the pixel intensity of the image resized to 17×16 grayscale. The hash captures relative brightness gradients — robust to small color shifts but sensitive to compositional changes.
Cluster Membership (Union-Find)

$$ \text{cluster}(I_a, I_b) = \mathbb{1}\big[ \text{Hamming}(H(I_a), H(I_b)) \leq 12 \big] $$

Two images are merged into the same cluster if their hashes differ in ≤ 12 of 256 bits. We restrict comparisons to within the same class (no cross-class clustering) since visually similar images of different classes are not duplicates.

Outcome

After regrouping, accuracy dropped from 99.73 % to a more honest 94.85 % — but for the first time, the focus lift began responding to attention supervision instead of being stuck at random. The "drop" is the point at which the project actually started measuring generalization rather than memorization.

Section 05

The Augmentation Pipeline

Augmentation is performed entirely on GPU using Kornia, with one pass per batch. The pipeline serves two purposes: regularization (preventing overfitting) and active background bias suppression (forcing the model to look at the fruit).

The 8-step AggressiveAug pipeline

01
Mask Edge Erosion · p = 0.30

Random binary erosion with kernel size ∈ {3, 5, 7}. Prevents the model from memorizing exact mask-edge shapes as a class signature.

02
Background Replacement · p = 0.95

The fruit pixels (selected by SAM mask) are composited onto a random kitchen/market background sampled from the GPU-resident bank. Forces the model to be invariant to whatever lies behind the fruit.

03
Background Blackout · p = 0.20

The background is set to pure black, leaving only the fruit visible. Confronts the model with the question: "can you classify based on fruit alone?"

04
Localized CLAHE · p = 0.70

Contrast Limited Adaptive Histogram Equalization, applied only inside the fruit mask. Enhances fine skin texture and pore detail (the discriminative signal for Formalin) without amplifying background noise.

05
Spatial Transforms · p = 0.4 each

Sequential horizontal flip → vertical flip → rotation (±30°) → affine (translate ±10%, scale 0.85–1.15) → perspective (distortion 0.3). Each independent.

06
Constrained Color Jitter · p = 0.7

Brightness ±20%, contrast ±20%, saturation ±20%, hue ±0.05 only. Hue is constrained because excessive shifts would destroy the subtle color indicators of formalin treatment and decay.

07
RandomErasing · p = 0.35

Random rectangular regions (5–20% of area) are erased to a random color. Forces the model to make decisions from partial views.

08
CutMix / MixUp · activates at epoch 5

Sample-mixing operations introduced after warmup, with α ramping from 0.1 to 0.5 over epochs 5–15. Smooths decision boundaries and improves calibration.

CutMix Mixed Loss

$$ \mathcal{L}_{\text{cutmix}} = \lambda \cdot \mathcal{L}_{CE}(y, y_a) + (1-\lambda) \cdot \mathcal{L}_{CE}(y, y_b) $$

Where \(\lambda\) is the area ratio of the unmixed region. Final image is constructed by pasting a rectangular crop from sample \(b\) onto sample \(a\).

Image normalization (ImageNet mean and std) is applied last, after all spatial and color transformations. The whole pipeline runs in ~150 ms per batch of 32 on the RTX 3070.

Section 06

Three Hybrid Architectures

We deliberately picked three architectures with different design philosophies to compare the tradeoffs between accuracy, interpretability, and deployment readiness.

Model A · EfficientNet+SwinT (Parallel Late Fusion)

The accuracy-first model. Two ImageNet-pretrained backbones (EfficientNetV2-S + SwinV2-Tiny) process the input in parallel. Their global-pooled features are concatenated and passed through a 2-layer MLP head to produce 15 logits.

Why this design: EfficientNetV2's MBConv blocks excel at local texture extraction; SwinV2's shifted-window attention captures global spatial relationships. Combining them gives the model both views simultaneously. Total parameters: 49 M.

Model B · DBA-ViNet (Dual-Branch Attention-Guided)

The interpretability-focused model. Based on Srinivasan et al. 2025 (BMC Plant Biology). One EfficientNetV2-B1 backbone produces both mid-level and high-level features. A learned attention gate over the high-level features acts as a mask on the mid-level features before they enter a small transformer (4 blocks, 192 dim, 4 heads). A cross-attention fusion combines local transformer output with global pooled features.

Model C · MobileNetV4-Hybrid-Large

The deployment model. Google's MobileNetV4 architecture (ECCV 2024) built on Universal Inverted Bottleneck (UIB) blocks + Mobile MQA attention. UIB unifies five micro-architectures (IB, ConvNext, FFN, ExtraDW, half-IB) into one searchable block. Mobile MQA shares a single Key/Value across all attention heads — a 39% speedup over standard multi-head attention with no accuracy loss.

Why this design: Mobile MQA produces unusually concentrated attention because the shared Key/Value forces all heads to agree on what's important — a side-effect that translates into much higher interpretability under Path W supervision. Total parameters: 37 M. Pretrained at 384×384 on ImageNet.

Mobile MQA (Multi-Query Attention)

$$ \text{MQA}(X) = \text{Concat}_{h=1..H}\big(\text{softmax}(Q_h K^\top / \sqrt{d_k}) V \big) $$

Where each head \(h\) has its own query \(Q_h\), but Key \(K\) and Value \(V\) are shared across all heads. This collapses the parameter count of attention by ~33% while empirically losing nothing in accuracy.

Loss function

The loss function combines soft cross-entropy (for CutMix/MixUp soft labels), label smoothing, and the Path W attention loss with epoch-dependent λ:

Combined Training Loss

$$ \mathcal{L}_{\text{total}} = -\sum_{c=1}^{15} \tilde{y}_c \log\,\text{softmax}(z_c) + \lambda(t) \cdot \mathcal{L}_{\text{Path W}} $$

$$ \tilde{y}_c = (1 - \epsilon)\, y_c + \frac{\epsilon}{C} $$

Where \(\tilde{y}_c\) is the smoothed soft label (\(\epsilon = 0.1\)), \(y_c\) is the one-hot or CutMix-mixed target, \(z_c\) is the logit, and \(\lambda(t)\) is the time-varying attention loss weight (see section 08).

Section 08 · Original contribution

Path W: Soft Mask Attention Supervision

Path W is the technique we developed to push focus lift above 1.0× (random) without hurting test accuracy. It supervises each model's spatial attention with the SAM-generated fruit mask, but as a soft regularizer, not a hard constraint.

The core idea

We don't force the model to look at fruit. We add a small loss penalty proportional to how much of the model's attention falls outside the fruit mask. The model is free to look anywhere — but pays a tax for looking off-target.

Outside-Mask Penalty

$$ \mathcal{L}_{\text{Path W}} = \frac{1}{B}\sum_{b=1}^{B} \sum_{i,j} \frac{A_{b,i,j}}{\sum_{i',j'} A_{b,i',j'}} \cdot (1 - M_{b,i,j}) $$

Where \(A\) is the model's spatial attention map (a class-aware CAM for Models A and C, or the explicit attention gate for Model B), normalized to sum to 1 per sample. \(M\) is the binary fruit mask. The product is summed over outside-mask pixels. Lower is better — zero means all attention falls inside the fruit.

Lambda schedule

The weight \(\lambda(t)\) is not constant. Path W is disabled for the first 3 epochs (warmup), then ramps linearly to its final value over 5 epochs. This lets the model first learn classification basics before being pushed to focus on fruit specifically.

Lambda Schedule

$$ \lambda(t) = \begin{cases} 0 & t < 3 \\ \lambda_{\max} \cdot \min(1, \frac{t-3}{5}) & t \geq 3 \end{cases} $$

Where \(t\) is the current epoch and \(\lambda_{\max} = 0.05\) for Models A and C (CAM-based supervision), or \(0.10\) for Model B (architectural attention gate). Model B uses double weight because supervising an explicit single-channel sigmoid gate can take stronger pull without destabilizing learned features.

How attention is extracted per model

  • Model A (EffNet+SwinT): The classifier weights of the head layer are projected back through the fc1 layer to produce a class-aware CAM weight on the EfficientNet feature map. This is a single matmul — no second backward pass.
  • Model B (DBA-ViNet): The attention gate output is supervised directly. It is a 1-channel sigmoid map, so the supervision is exact, not derived.
  • Model C (MobileNetV4): Same CAM strategy as Model A, applied to the final UIB block features.

Why this works

Path W is a structured prior. It tells the model where to look without telling it what features to find. Given that ImageNet-pretrained backbones already produce strong visual features, the only thing they lack is the bias toward looking at the actual subject. Path W adds exactly that bias — and our results show it doesn't hurt accuracy: MobileNetV4 jumped from 1.46× to 4.62× focus lift, with test accuracy staying within 2 points of the unrestricted baseline.

Section 09

Evaluation Methodology

Evaluation goes beyond standard accuracy because the project's success criteria (Section 7 of the project report) demand interpretability and background invariance, not just classification correctness.

Standard metrics

  • Test accuracy: fraction of test samples correctly classified.
  • Per-class precision, recall, F1: with special attention to Formalin classes.
  • Confusion matrix: reveals exactly where each model fails (Fresh ↔ Formalin is the dominant axis).
  • Mean confidence: average max softmax probability — a lower number indicates better calibration.

XAI metrics (the harder part)

Focus Score (per image)

$$ \text{focus}(I) = \frac{\sum_{i,j} A_{i,j} \cdot M_{i,j}}{\sum_{i,j} A_{i,j} + \varepsilon} $$

Fraction of total Grad-CAM energy that falls inside the fruit mask. \(A\) is the heatmap, \(M\) the binary mask. \(\varepsilon = 10^{-9}\) to avoid division by zero.
Focus Lift (aggregate)

$$ \text{lift} = \frac{\overline{\text{focus}(I)}}{\overline{\text{mask\_area}(I)\,/\,\text{img\_area}}} $$

Mean focus score divided by mean mask-area ratio. A lift of 1.0 means random — the model attends to fruit no more than chance would predict given its size in the image. Lift > 1.8 is considered "trustworthy" in XAI literature.

Stress tests

  • Background invariance test: 100 test images composited onto random unfamiliar backgrounds (markets, kitchens, outdoor scenes). Models must maintain accuracy > 90% per the project report's Section 7 criteria.
  • Inference latency: single-image and batch-32 forward pass times under AMP, measured on the deployment hardware.
  • Visual XAI inspection: Grad-CAM heatmaps for one sample per class are visually inspected to verify attention falls on fruit-skin texture, not background or shape.

Section 10

Explainable AI Implementation

We use Grad-CAM from the pytorch-grad-cam library, with carefully selected target layers per model. The choice of target layer significantly affects the produced heatmap.

Target layer selection

ModelTarget LayerWhy
EffNet+SwinT effnet.blocks[-1] Last MBConv block in the EfficientNet branch — captures rich texture features before final pooling
DBA-ViNet global_backbone.blocks[-1] Last block of the global EfficientNetV2-B1 backbone — same rationale
MobileNetV4 backbone.blocks[-1] Last UIB stage just before pooling — preserves fine spatial detail

Grad-CAM formulation

Grad-CAM Heatmap

$$ L^c_{\text{Grad-CAM}} = \text{ReLU}\left(\sum_k \alpha^c_k A^k\right) $$

$$ \alpha^c_k = \frac{1}{Z}\sum_{i,j} \frac{\partial y^c}{\partial A^k_{i,j}} $$

For predicted class \(c\), \(\alpha^c_k\) is the global-pooled gradient of the class score with respect to feature map \(A^k\), and \(Z\) is the spatial size. The ReLU keeps only positive influences. The result is a 8×8 spatial map at the last convolutional layer, upsampled to 256×256 for visualization.

Interpretation pipeline

  1. Run forward pass to get predicted class \(c\)
  2. Backward pass to compute \(\partial y^c / \partial A^k\) for each feature map \(k\)
  3. Global-average-pool the gradients to get channel weights \(\alpha^c_k\)
  4. Linear combination of feature maps weighted by gradients
  5. ReLU to discard negative contributions
  6. Bilinear upsample to input resolution
  7. Normalize to [0, 1] and overlay on original image

Section 11

Deployment

The web demo serves all three models from a single FastAPI Python backend. Models are loaded into memory once at startup and held in memory for the entire server lifetime. Requests are served synchronously: preprocess → model A inference → model B → model C → 3 Grad-CAMs → response.

Display logic

The frontend applies these confidence-based display rules:

  • If top-1 confidence ≥ 75% → show only top-1 (model is confident)
  • If top-1 confidence ∈ [35%, 75%) → show top-1 AND top-2
  • If top-1 confidence < 35% → display "Object Not Detected"
  • If top-2 confidence < 35% (when shown) → display "Object Not Detected" for second slot

Why this approach

  • Single binary deployment: one Python process, one port, one set of files.
  • No model server: avoids TorchServe complexity for a 3-model demo.
  • Static frontend: served by the same FastAPI process via StaticFiles.
  • CORS enabled: permissive in dev, can be tightened for production.
